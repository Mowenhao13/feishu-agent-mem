## 错误处理

错误处理逻辑：
```
工具调用 -> 返回错误 -> 错误处理反馈LLM -> LLM自我纠正 -> 修正后重试
```

隐形失败三种情况：
- 死循环：反复调用同一工具
- 方向偏离：解决了错误的问题
- 上下文溢出：丢失早期关键信息

自我反思机制：
```
执行N步 -> 暂停反思 -> 对齐原始目标 -> 继续执行 -> 回滚 + 重新规划
```
检查点机制（别从头再来）：
```
Step1(已保存) -> Step2(已保存) -> Step3(检查点) -> Step4(已保存) -> Step5(失败)
```
兜底策略：快速失败+部分返回
1. 最大循环次数 防止死循环
2. 最大Token预算 控制成本
3. 超时熔断 避免无限等待
4. 人工介入 最终兜底

失败恢复机制：
```
失败检测 -> 显性/隐性分类 -> 自我纠正/回滚 -> 护栏检查 -> 降级/人工介入
```

## 系统提示词

设计一：静态段 vs 动态段
静态段
- 身份介绍
- 安全指令
- 代码风格检测
- 工具使用指南
- 行为风险评估框架
动态段
- 当前会话使用工具
- 记忆系统索引内容
- 工作目录和操作系统信息
- 用户语言偏好
- 外部插件使用说明

服务端复用计算结果（通用规则放前面，特定信息放后面，提高缓存命中率）：
```
用户A请求 -> 静态段（通用）跨用户共享缓存 -> 动态段A 用户A特有信息 用户B请求
```

设计二：
代码风格指令极其具体：
- 规则具体
- 可执行，有明确边界
具体规则演示：
- 不要给你没改过的代码加注释，文档字符串或类型标注，只在逻辑不自洽的地方加注释
- 不要为假设的未来需求做设计，三行相似代码比一个过早的抽象好
- 不要加你没被要求的功能，bug修复不需要顺便清理周围的代码
- 不要加错误处理应对不可能发生的场景，信任内部代码，只在系统边界做校验

## Agent工具设计
原则一：在万能工具里写「别用我」
专用工具价值：可控边界
- 精确控制输出格式
- 文件缓存去重
- Token 预算检查
在通用工具描述里显示写出：那些场景该用其他工具，不靠模型判断

原则二：两级工具加载
例如：
```
模型需要 延迟工具 -> 调用Tool Search 传入关键词 -> 匹配searchHint+工具名+说明 -> 返回完整定义 含参数格式 -> 模型正常调用
```
原则三：工具输出要有预算
```
大部分工具（10w字符） 搜索工具（2w字符） 文件读取（特殊处理）
```
